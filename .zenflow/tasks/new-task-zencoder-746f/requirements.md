# PRD: Multi-Prong Research Distillation System

## 1. Problem Statement

The project has a large research corpus (38 Perplexity-generated research files, 26 overview files, Kimi-Research artifacts, CSV datasets, and prior research threads) covering autonomous AI coding systems. This raw research is spread across 12 taxonomy layers and multiple source formats. It cannot be directly consumed by product builders who need to create agent skills, modes, workflows, prompts, rules, MCP configurations, context strategies, task decomposition patterns, workspace management specs, and situational technique playbooks.

There is no system to extract, deduplicate, rank, tag, cross-reference, and assign knowledge from this corpus to the specific product artifacts that the multi-agent autonomous coding platform requires.

## 2. Objective

Build a distillation pipeline that processes the entire research corpus through four complementary prongs, producing:

1. A **Master Knowledge Atom Registry** (Prong 1) — deduplicated, ranked, tagged knowledge atoms extracted from all research files
2. **Domain Groupings** (Prong 2) — atoms organized by 12 technical domains with cross-domain links and gap analysis
3. **SDLC Phase Mappings** (Prong 3) — atoms placed into 8 development lifecycle phases with sequenced techniques and transitions
4. **Product Specs** (Prong 4) — atoms assembled into YAML spec templates for 10 product categories

Each knowledge atom is defined once with a unique ID and referenced by ID across all prongs. No content duplication.

## 4. Knowledge Atom Specification

### 4.1 Atom Types
| Type | Definition |
|------|-----------|
| TECHNIQUE | A named method with enough detail to execute |
| METRIC | A specific measurement that constrains or informs a decision |
| CONSTRAINT | A hard limit or invariant the system must respect |
| TOOL | A specific tool/platform with evaluated capabilities |
| COMBINATION | How multiple techniques/tools work together |
| FAILURE_MODE | A documented way things break, with detection and response |
| ANTI_PATTERN | Something that seems right but is demonstrably wrong |
| TRADEOFF | A genuine choice between approaches with criteria for deciding |
| RECIPE | An ordered sequence of steps for accomplishing a specific task |

### 4.2 Atom Schema
Each atom must have:
- `ID`: Unique identifier (e.g., KA-047)
- `TYPE`: One of the 9 types above
- `CONTENT`: The actual knowledge, 1-5 sentences max
- `EVIDENCE_STRENGTH`: STRONG (verifiable source + specific data) | MODERATE (plausible but unverified numbers) | WEAK (assertion without data)
- `SOURCE`: Research file path(s) where found
- `DOMAINS`: Which technical domains this applies to (D1-D12)
- `SDLC_PHASES`: Which lifecycle phases this applies to (P1-P8)
- `PRODUCTS`: Which product types this feeds (PC1-PC10)

### 4.3 Extraction Rules

**KEEP** if it contains:
1. A specific technique with enough detail to implement or evaluate
2. A specific number, benchmark, or threshold that constrains a decision
3. A specific tool with evaluated capabilities (not just a name-drop)
4. A combination recipe showing how multiple approaches work together and in what order
5. A documented failure mode with detection method AND response action
6. A ranked tradeoff with decision criteria (not just "A vs B")
7. A hard constraint with evidence for why

**DISCARD** if it is:
1. Definition of well-known concepts
2. Technique names without procedures
3. Unverifiable claims with no actual reference
4. Generic advice ("Handle errors appropriately")
5. Structural scaffolding (navigation links, "See also")
6. Prior Research Integration sections that admit inability to access sources
7. Low-confidence padding (topics marked LOW with <3 bullets)
8. Vendor marketing without comparative data
9. Aspirational statements ("Future work should explore...")
10. Research methodology descriptions

### 4.4 Deduplication Rules
When the same knowledge appears in multiple files:
- Keep version with most specific numbers
- Keep version with most retrievable citation
- Keep version with most implementation detail
- Merge complementary details into one atom
- Record all source files in SOURCE field

### 4.5 Ranking Rules
Within each TYPE, rank by:
1. Evidence strength (STRONG > MODERATE > WEAK)
2. Specificity (concrete procedure > vague method > name-only)
3. Uniqueness to AI agent systems (agent-specific > general SE)

## 8. Output Artifacts

### 8.1 Master Knowledge Atom Registry
- Format: Structured data (JSON or YAML)
- Contains all extracted atoms with full tagging
- Deduplicated, ranked within each TYPE

### 8.2 Domain Specs (12 files)
- One file per domain (D1-D12)
- References atoms by ID
- Includes gap analysis

### 8.3 Phase Specs (8 files)
- One file per SDLC phase (P1-P8)
- References atoms by ID
- Includes transition conditions

### 8.4 Product Specs (variable count)
- YAML files using the 10 spec templates (Mode, Skill, Workflow, Rule, Context Strategy, Technique, Task Decomposition Pattern, MCP Configuration, Prompt)
- Each spec references consumed atom IDs
- Includes confidence level and gaps

### 8.5 Cross-Reference Validation Report
Verifies:
- Every atom referenced by at least one domain, phase, and product
- Skills referenced by Modes/Workflows exist as Skill specs
- Context strategies referenced by Modes exist
- Rules consistently applied
- MCP configurations match Mode tool lists
- No orphan atoms

### 8.6 Gap Report
- Strong backing (>=5 STRONG atoms)
- Adequate backing (2-4 atoms, mixed evidence)
- Weak/no backing (<2 atoms or WEAK only)
- Orphan research (atoms with no product home)
- Contradictions (atoms that disagree)

## 9. Quality Gates

All outputs must satisfy:
- No textbook padding — every atom is AI-agent-specific, not general SE
- No unranked lists — techniques always ordered by evidence, effectiveness, simplicity
- No vague procedures — steps name specific tools/actions, not "analyze the code"
- Combination recipes are step-by-step with numbered sequences
- Every constraint has enforcement (detection + response)
- Every failure mode has a response (detection + mitigation)
- Cost/token awareness in skills and strategies
- No duplicate content — each atom defined once, referenced many times
- Specs are self-contained — an agent can follow a spec without reading original research
- Gaps are explicit — missing info called out, not filled with generalities

## 10. Anti-Patterns (Must Not Produce)

- Research summaries or catalogs
- Unranked technique lists
- "Consider using X" without conditions
- Background explanations of well-known concepts
- Padding of thin areas with generic content
- Invented procedures where research only says "X is effective" but not HOW
- Duplicated content across prongs (define once, reference by ID)

## 11. Scope & Assumptions

### 11.1 In Scope
- Processing all files in `docs/research/`, `Kimi-Research/`, and root-level research files
- Extracting knowledge atoms from all source formats (Markdown, CSV data for reference)
- Producing all four prong outputs plus validation and gap reports
- Using existing YAML spec templates defined in the task description

### 11.2 Out of Scope
- Executing the research prompts (already done — 38 files exist)
- Modifying existing research files
- Building runtime agent infrastructure
- Implementing the agents themselves

### 11.3 Assumptions
- All 38 Perplexity research files are complete and available (confirmed)
- All 26 overview files are available (confirmed)
- Kimi-Research corpus is available (confirmed)
- The distillation is a batch processing task — not a live/streaming system
- Output format is files on disk (YAML, Markdown, JSON) — no database required
- A Python-based processing pipeline is appropriate given existing `scripts/` infrastructure uses Python

## 12. Success Criteria

1. Every research file in the corpus has been processed
2. Knowledge atoms are extracted, deduplicated, and ranked
3. All four prong outputs are produced with correct cross-references
4. Validation report shows no orphan atoms and no broken references
5. Gap report identifies areas needing additional research
6. Product specs are in YAML format using the defined templates
7. Quality gates pass for all output artifacts
